1.  EXECUTIVE SUMMARY
================================================================================

ARGUS (Agentic Research & Governance Unified System) is one of the most
architecturally sophisticated multi-agent debate frameworks in existence. It
implements a Conceptual Debate Graph (C-DAG), Bayesian belief propagation,
hybrid BM25 + FAISS retrieval, provenance hash-chain tracking, and a suite of
novel evaluation metrics including ARCIS, EVID-Q, DIALEC, REBUT-F, CONV-S,
PROV-I, CALIB-M, and EIG-U. Every one of these capabilities requires deep
technical knowledge to configure and operate. The framework speaks fluently to
engineers and researchers. It speaks not at all to the vast majority of people
with a legitimate intellectual need to evaluate complex claims.

ARISTOTLE — the Autonomous Reasoning Intelligence for Structured
Topic-Orchestrated Logical Engagement — is the complete answer to that gap.

Two architectural decisions define this edition of the ARISTOTLE protocol:

  (1) MODULE PLACEMENT: ARISTOTLE is not a separate system, external wrapper,
      or standalone application. It is a first-class sub-module of the
      argus-debate-ai package, living at argus.aristotle, installed and
      imported alongside every other ARGUS component.

  (2) SINGLE-PANE SPLIT INTERFACE: ARISTOTLE does not use two separate
      Streamlit tabs or screens. The entire experience — chat with ARISTOTLE
      on the left, live debate visualization (including the Full Lifecycle DAG)
      on the right — lives in one unified, split-pane Streamlit interface.
      The right panel updates continuously as the debate progresses. The user
      never changes tabs. Everything is visible simultaneously.

The core proposition: A user opens ARGUS, types a question in the left chat
pane, and watches the Full Lifecycle DAG grow on the right as ARISTOTLE builds
and runs the debate autonomously. Status updates, belief trajectory, evidence
heatmap, and all other visualization panels update in real time within the same
view. One sentence in; complete, visualized, auditable truth out.

3.  THE PROBLEM SPACE: OLD WORKFLOW vs. ARISTOTLE PROTOCOL
================================================================================

3.1  THE OLD ARGUS WORKFLOW (Expert-Only)
─────────────────────────────────────────

Every ARGUS session currently requires the user to make at least seven
expert-level decisions before a single debate token is generated.

  DECISION POINT                  WHAT THE USER MUST KNOW
  ──────────────────────────      ──────────────────────────────────────────
  1. Proposition formulation      Restate question as falsifiable claim with
                                  an explicit prior probability (0.0-1.0)

  2. Specialist configuration     Agent count, domain focus, retrieval
                                  sources, epistemic priors for each agent

  3. Refuter setup                Refuter count, intensity, rebuttal strategy,
                                  and target thresholds

  4. Jury architecture            Aggregation method (Bayesian vs. majority),
                                  jury size, calibration parameters

  5. Stopping criteria            Max rounds, convergence thresholds, budget
                                  limits, information gain cutoffs

  6. Retrieval configuration      BM25/FAISS balance, source list, embedding
                                  model selection

  7. Output interpretation        Read Bayesian posterior curves, interpret
                                  ARCIS/EVID-Q/DIALEC, navigate C-DAG manually

Each is a potential failure point for a non-technical user. Even for expert
users, each decision consumes time and attention that could be spent
interpreting results rather than configuring systems.

3.2  THE ARISTOTLE PROTOCOL (Any User)
───────────────────────────────────────

Every one of these seven decision points is eliminated. ARISTOTLE owns them
all. The user's only action is typing a question in the left chat pane.

  USER INPUT (the entirety of what is required):
  "Does intermittent fasting actually work for long-term weight loss,
   or is the evidence overhyped?"

From this, ARISTOTLE executes the complete workflow autonomously across five
layers — while the user watches the right visualization pane update live.

  STEP    WHAT ARISTOTLE DOES                     WHAT USER SEES
  ──────  ──────────────────────────────────────  ────────────────────────────
  L1      Parses query: identifies domain,         Left pane: framing narrative
          controversy, debate type, prior          appears in chat as ARISTOTLE
                                                   thinks out loud

  L2      Builds agent roster: specialist          Left pane: Topology Preview
          personas, refuter config, jury type,     Card appears; user can
          round count, cost estimate               approve or adjust via chat

  L3      Runs debate autonomously through          Right pane: Full Lifecycle
          ARGUS's existing RDC machinery;           DAG begins populating;
          monitors and steers mid-debate            all panels update live

  L4      Streams events to the right pane          Right pane: nodes appear
          visualization panels in real time         round by round; status bar
                                                    shows current posterior

  L5      Synthesizes plain-language verdict        Left pane: complete result
          and delivers it in chat                   package delivered in chat

  (CLI command added to argus's existing CLI system)

5.  ARISTOTLE ARCHITECTURE — FIVE LAYERS
================================================================================

ARISTOTLE is a five-layer meta-orchestration stack. Each layer is a distinct
module inside argus.aristotle and each feeds sequentially into the next.

  LAYER  MODULE           FUNCTION                           KEY OUTPUT
  ─────  ───────────────  ─────────────────────────────────  ──────────────────
  L1     framing.py       Parse NL query into Debate Frame   DebateFrame JSON
  L2     topology.py      Build agent ecosystem from Frame   TopologySpec JSON
  L3     monitor.py       Drive ARGUS; monitor mid-debate    C-DAG + event log
  L4     interface.py     Single-pane split UI + DAG viz     Live Streamlit UI
  L5     synthesis.py     Plain-language result package      Chat result card

5.1  LAYER 1 — INTENT PARSING & FRAMING ENGINE (framing.py)
────────────────────────────────────────────────────────────

Activated when the user submits a message in the left chat pane. Performs
seven analytical operations using a lightweight fast-inference model (not the
full debate LLM — saves cost and latency):

  (a) DOMAIN CLASSIFICATION
      Identifies primary and secondary knowledge domains from the query.
      Medical claims activate PubMed-weighted retrieval. Legal claims invoke
      case law evidence standards. Technical claims use empirical criteria.
      This is semantic domain inference, not keyword matching.

  (b) DEBATE TYPE DETECTION
      Classifies into one of six types:
        Binary Factual   — "Is X true?"
        Causal           — "Does X cause Y?"
        Comparative      — "Is X better than Y?"
        Predictive       — "Will X happen?"
        Normative        — "Should X be done?"
        Definitional     — "What does X actually mean?"
      Each type has a different optimal agent configuration and evidence
      standard that L2 uses to build the topology.

  (c) CONTROVERSY LEVEL SCORING
      A 0.0-1.0 score computed from:
        - Presence of hedging language in query ("actually," "really,"
          "overhyped," "just a," "or is it")
        - Fast web probe counting conflicting expert positions
        - Domain-specific controversy priors (nutrition science = inherently
          contested; basic Newtonian mechanics = not contested)

  (d) EVIDENCE DENSITY ESTIMATION
      Estimates whether the topic has rich empirical evidence (published
      trials, datasets, replication studies) or sparse evidence (emerging
      technology, speculative forecasts). Determines whether the upcoming
      debate will be evidence-intensive or inference-intensive — which
      affects round count estimation and retrieval configuration.

  (e) STANCE SPACE MAPPING
      Identifies the full range of legitimate positions on the topic — not
      just binary pro/con, but the multi-dimensional space of actual expert
      disagreement. A climate policy debate has at least four distinct
      positions that differ in kind, not just degree.

  (f) CLAIM DECOMPOSITION
      Multi-part or complex queries are broken into atomic sub-propositions
      that can be debated in parallel or in sequential dependent rounds.
      The primary claim is identified as the debate's anchor; sub-claims
      are scheduled as dependent rounds in L3's execution plan.

  (g) PRIOR PROBABILITY CALIBRATION
      The Bayesian prior for each proposition is set using a weighted
      combination of:
        - Base rate for claims of that type in that domain
        - Rapid n=10 literature scan with conclusion direction scoring
          (weighted by recency and study type: RCT > cohort > case study)
        - Explicit uncertainty language in the user's query

5.2  LAYER 2 — DYNAMIC TOPOLOGY BUILDER (topology.py)
───────────────────────────────────────────────────────

The most novel layer. Takes the DebateFrame from L1 and constructs the
entire agent ecosystem from scratch. No templates. No pre-defined configs.
Every agent in every debate is invented specifically for that topic.

  SPECIALIST AGENT CONSTRUCTION
  For each specialist, topology.py generates a full AgentSpec:

    - Agent Name: a realistic human name that signals expertise to the LLM
      and to the user (e.g., Dr. Amara Osei, Prof. Sarah Chen)
    - Domain Mandate: precise description of what aspect of the evidence
      landscape this agent owns
    - Evidence Source Priority: which retrieval sources to weight highest
      (PubMed, arXiv, legal databases, financial APIs, general web)
    - Epistemic Prior: where this agent starts in belief-space — set to
      ensure meaningful initial disagreement following Wu et al. 2025's
      finding that "initial mild disagreement increases the likelihood of
      post-debate improvement"
    - Evidence Type Focus: empirical studies, meta-analyses, case reports,
      expert consensus statements, etc.

  SPECIALIST COUNT SCALING
    Scales with topic domain breadth × controversy score:
      Controversy < 0.3  →  2 specialists
      Controversy 0.3-0.5 → 3 specialists
      Controversy 0.5-0.7 → 4 specialists
      Controversy 0.7-1.0 → 5-6 specialists

  REFUTER CONFIGURATION
    Refuters are assigned opposing epistemic priors to specialists and
    instructed to attack methodology, not just conclusions. Intensity
    scales to controversy score:
      LOW     — polite challenge; appropriate for near-settled science
      MEDIUM  — active rebuttal; appropriate for contested empirical claims
      HIGH    — adversarial cross-examination; appropriate for highly
                controversial or values-laden disputes

  JURY ARCHITECTURE SELECTION
    topology.py selects one of three jury architectures based on debate type:

      Bayesian Jury (3 members)
        Used for: empirical factual and causal claims
        Mechanism: members vote with weighted confidence; ARGUS's existing
        Bayesian aggregation runs over their outputs

      Deliberative Jury (5 members)
        Used for: comparative and normative claims
        Mechanism: members debate intermediate verdicts before aggregating,
        following the ReConcile protocol

      Validity-Weighted Jury (5 members)
        Used for: contested scientific claims
        Mechanism: following Wu et al. 2025's finding that "weighting update
        decisions by argument validity maximizes post-debate improvement,"
        each juror's influence is proportional to assessed reasoning quality

  ROUND COUNT AND BUDGET ESTIMATION
    Computed as:
      Base Rounds (2)
      + Controversy Bonus (0-3 based on controversy score)
      + Evidence Density Bonus (0-2 if landscape is rich)
      + Sub-Claim Adjustment (-1 per parallel sub-claim handled in batch)
      Capped at 8 rounds maximum; floored at 2 rounds minimum

    Pre-run cost estimate is computed from:
      round_count × agent_count × expected_tokens_per_agent_per_round
    Displayed in the left chat pane before the user confirms launch.

5.3  LAYER 3 — AUTONOMOUS EXECUTION ENGINE (monitor.py)
────────────────────────────────────────────────────────

Once the user confirms the topology in the chat, monitor.py hands the
TopologySpec to ARGUS's existing RDCOrchestrator and runs a parallel
meta-monitor thread that watches the debate's information-theoretic
trajectory in real time, intervening when necessary.

  MONITORING MECHANISM   TRIGGER                    INTERVENTION
  ────────────────────   ─────────────────────────  ────────────────────────
  Convergence Watch      Posterior stable < 0.02    Early termination;
                         for 2 consecutive rounds   jury verdict triggered;
                                                    gold star appears in DAG

  Stagnation Detector    2 rounds produce zero      Inject provocateur prompt
                         EVID-Q > 0.4 evidence      forcing adjacent domain
                                                    search

  Drift Correction       Semantic similarity to     Refocus directive issued
                         proposition drops < 0.70   to all agents; yellow
                                                    edge in DAG highlights
                                                    drift path

  Confidence Escalation  Agent self-confidence      Hide confidence from all
  Guard                  exceeds 0.90 in rounds     agents; trigger external
                         1-3 (anti-Bayesian drift   calibration reset
                         failure mode)

  Evidence Drought       Specialist retrieves < 3   Replace with backup agent
                         unique nodes in a round    from backup_agents.py;
                                                    agent swap shown in DAG

  Budget Overflow        Running token count        Accelerate: summative
                         exceeds 90% of cap         evidence only; skip
                                                    sub-claim rounds; amber
                                                    warning in status bar

  Deadlock Detection     Jury split 50/50 with      ARISTOTLE submits a
                         high confidence for 2+     structured meta-argument
                         rounds                     citing highest-quality
                                                    evidence from both sides;
                                                    ARISTOTLE node appears
                                                    in DAG

Every intervention is logged as an ORCHESTRATOR_DECISION event in the
ProvenanceLedger (via decision_log.py). Every intervention is visible in
the right visualization pane as a white pulsing marker in the Full
Lifecycle DAG and as an entry in the Timeline sub-panel.

5.4  LAYER 4 — SINGLE-PANE SPLIT INTERFACE (interface.py + dag_viz.py)
───────────────────────────────────────────────────────────────────────

The user-facing interface. Described in full detail in Section 6 and 7.
The key architectural points at the layer level:

  - Built as a Streamlit application (aristotle_app.py) added to argus's
    existing application suite
  - Single-pane split layout: left column (chat, 38% width) and right
    column (visualization, 62% width) rendered simultaneously in one page
  - No tabs, no page navigation, no separate browser windows
  - Right panel updates via st.session_state event queue + Streamlit
    re-run cycle at 800ms intervals as debate events arrive
  - All visualization built with Plotly figures rendered via
    st.plotly_chart(use_container_width=True)

5.5  LAYER 5 — OUTPUT SYNTHESIS (synthesis.py)
───────────────────────────────────────────────

Transforms the structured debate record into a complete result package
delivered in the left chat pane after the debate concludes. Synthesis is
strictly grounded — synthesis.py only receives structured C-DAG nodes and
ProvenanceLedger records as context, never free-form debate transcripts.
Every narrative sentence must map to a specific C-DAG node ID.

  OUTPUT COMPONENT              CONTENT AND GROUNDING
  ─────────────────────────     ─────────────────────────────────────────────
  The Verdict                   Plain-English statement of jury finding with
                                confidence in natural language:
                                  "Strong Evidence Supports"
                                  "Contested with Moderate Evidence"
                                  "Insufficient Evidence to Conclude"
                                  "Evidence Actively Against"
                                Grounded in: jury posterior + CONV-S score

  The Reasoning Narrative       3-5 paragraph account of how the debate
                                unfolded: key arguments, decisive evidence,
                                sharpest disagreements, minority view
                                Grounded in: C-DAG highest-weight argument
                                chains

  The Dissent Log               If jury members disagreed: minority position
                                surfaced and explained in plain language;
                                what evidence would have reversed the majority
                                Grounded in: jury deliberation records +
                                C-DAG counter-evidence nodes

  What Could Change This        2-3 specific research suggestions derived from
                                the VoI Planner's highest-EIG unexplored
                                evidence nodes — a plain-language research
                                agenda for the user
                                Grounded in: VoI Planner EIG scores

  Evidence Provenance Map       Every claim hyperlinked to its source evidence
                                node in the C-DAG, which links to original
                                source citation; full PROV-O compliance,
                                SHA-256 hash-chain integrity
                                Grounded in: ProvenanceLedger

  Metric Dashboard              All ARGUS-native metrics alongside plain-
                                language interpretations:
                                "The arguments were highly coherent (ARCIS=0.87)"
                                Grounded in: ARGUS evaluation framework

6.  THE SINGLE-PANE SPLIT INTERFACE — COMPLETE SPECIFICATION

6.1  CORE DESIGN PRINCIPLE
────────────────────────────

The ARISTOTLE interface is a single Streamlit page. There are no tabs. There
are no separate screens. There is no navigation. The user sees everything
simultaneously: their conversation with ARISTOTLE on the left, and the live
evolving debate visualization on the right.

This matters because:

  - A user who wants to ask "why did confidence spike in round 3?" should
    be able to look right, see the DAG, and look left to ask the question
    in the same eye movement — not navigate between tabs
  - Status updates (posterior probability, round number, convergence
    progress) should be visible at all times without any action from the user
  - The visualization does not require active user attention to update —
    it updates automatically while the user continues conversing on the left

6.6  REAL-TIME UPDATE MECHANISM
────────────────────────────────

The right pane achieves live updates through Streamlit's session state
and periodic re-run architecture:

  (a) The execution engine (L3/monitor.py) emits a structured JSON event
      for each debate action (evidence submitted, rebuttal filed, Bayesian
      update computed, ARISTOTLE intervention, round complete, verdict)

  (b) Each event is appended to st.session_state["event_queue"]

  (c) A background thread (running via concurrent.futures.ThreadPoolExecutor)
      processes events from the queue and updates the visualization data
      in st.session_state["dag_data"], ["belief_data"], ["heatmap_data"]

  (d) Streamlit's st.empty() containers hold the Plotly figures;
      the app re-runs on a 800ms st.rerun() cycle triggered by a
      st.session_state flag set by the background thread

  (e) On each re-run, each panel reads its data from session state and
      redraws only changed elements — additive updates, not full re-renders

  Chat messages on the left pane are also appended to session state
  by the monitoring thread and appear on the same re-run cycle.

7.3  TIERED LAYOUT — THE FIVE-TIER HIERARCHY
──────────────────────────────────────────────

The DAG uses a strict top-to-bottom layered layout with five tiers,
matching the reference image's visual structure:

  TIER 1 — PROPOSITION LAYER
    Single cyan diamond node at the top center.
    The epistemic anchor of the entire debate graph.
    Y-position: fixed at top of canvas.
    X-position: center.

  TIER 2 — SPECIALIST LAYER
    Blue square nodes spread evenly across Tier 2.
    X-positions: evenly spaced across the canvas width.
    Y-position: fixed below Tier 1.
    Grey dashed ASSIGNED edges connect from Proposition down to each
    Specialist with a brief draw animation when each agent is activated.

  TIER 3 — EVIDENCE LAYER
    Green and red circle nodes branch downward from their respective
    Specialist nodes. Support evidence nodes naturally cluster toward
    the center-bottom; attack evidence nodes spread to the outer edges.
    This creates the characteristic diamond/rhombus shape visible in
    the reference image — the full set of evidence nodes forms a
    roughly diamond-shaped spread across Tier 3.
    This tier grows continuously across all debate rounds.

  TIER 4 — REBUTTAL + BAYESIAN UPDATE LAYER
    Amber triangle nodes (Rebuttals) appear at the level of or slightly
    below the evidence they target.
    Lavender hollow circle nodes (Bayesian Updates) appear at the
    center of Tier 4 after each round completes, connected to all
    evidence nodes that contributed to that round's posterior update.

  TIER 5 — VERDICT LAYER
    Single gold star node at the bottom center.
    Connected from the final Bayesian Update node via a thick gold
    CONCLUDES edge.
    Y-position: fixed at bottom of canvas.
    X-position: center.

  ROUND DIVIDERS
    A horizontal dashed line is drawn across the full width of the
    canvas after each round completes, labeled:
      "Round N complete — Posterior: X.XX"
    New nodes added in the next round appear below this line.
    This makes the round-by-round progression visually clear even as
    the graph grows dense.

8.1  BELIEF TRAJECTORY LINE CHART (Zone B, left)

Implementation: plotly.express.line with round number on X-axis and

What it shows: The epistemic heartbeat of the debate — how belief in

Convergence is visible as the line flattens. Evidence events produce

Non-technical annotation: Tooltips explain each inflection in plain

from 0.55 to 0.71."

8.2  EVIDENCE QUALITY HEATMAP (Zone B, right)
──────────────────────────────────────────────

Implementation: plotly.express.imshow. Rows are agents; columns are
evidence items submitted; cell color represents EVID-Q score
(yellow=low quality, orange=medium, dark red=high quality).

What it shows: At a glance, which agents are producing high-quality
evidence and which are producing noise. Rounds producing weak evidence
across all agents appear as entire yellow columns.

Live update: Each time an agent submits evidence, a new cell appears
in that agent's row with its EVID-Q color. Agent replacement events
cause a new row to appear in place of the replaced agent's row.

8.3  ARGUMENT FLOW SANKEY DIAGRAM (Zone C)
──────────────────────────────────────────

Implementation: go.Sankey. Flows: Proposition → Evidence → Rebuttal →
Counter-Rebuttal → Finding → Verdict. Flow width represents cumulative
weight in the C-DAG. Flows reaching the Verdict are gold; flows
neutralized by rebuttals terminate early.

What it shows: The genealogy of the verdict — which argument threads
survived all challenges, which were neutralized, and which finding
nodes have the most argumentative ancestry.

8.4  AGENT INFLUENCE RADAR CHART (Zone C)
──────────────────────────────────────────

Implementation: go.Scatterpolar. Five axes: Evidence Submitted, Average
EVID-Q, Rebuttal Success Rate, Rounds Active, Jury Influence Score.
Each agent gets its own colored polygon trace.

What it shows: The contribution profile of each agent across all debate
dimensions. A specialist with many low-quality submissions looks very
different from one with few but decisive high-quality submissions.

8.5  CALIBRATION RELIABILITY DIAGRAM (Zone C)
──────────────────────────────────────────────

Implementation: go.Bar (confidence bins 0.0-1.0) with go.Scatter
overlay (observed accuracy per bin from historical sessions).
Perfect calibration diagonal drawn for reference.

Why this matters: The documented anti-Bayesian confidence escalation
problem (arXiv:2505.19184, 2025) makes calibration monitoring essential.
When jury confidence systematically exceeds observed accuracy, the bars
pull away from the diagonal and ARISTOTLE flags this in the left chat.

8.6  DEBATE TIMELINE EVENT PLOT (Zone C)
──────────────────────────────────────────

Implementation: Horizontal Gantt-style timeline built with go.Bar.
Each round is a band; each event is a marker colored by event type.
ARISTOTLE interventions appear as white markers, visually distinct
from all normal debate events.

What it shows: The complete chronological narrative of the debate —
every event in order, with timestamps, agent identifiers, event type,
and the ARISTOTLE intervention log.

9.  RESEARCH GAPS — WHAT REMAINS UNSOLVED IN AGENTIC AI
================================================================================

9.1  STATIC AGENT TOPOLOGIES
──────────────────────────────

The overwhelming majority of multi-agent systems in 2025 operate with
pre-defined, fixed organizational structures. LangChain, CrewAI, AutoGen,
LangGraph, MetaGPT — every major framework requires the developer to
specify the agent team before execution. The DAI 2025 conference paper
"Orchestrating Human-AI Teams" explicitly identifies the Autonomous Manager
Agent as an open research challenge: "task decomposition, dynamic resource
allocation, and adaptive re-planning remains fundamentally challenging."

ARISTOTLE's topology.py — which constructs the entire agent ecosystem
dynamically from a single NL query — is a direct answer to this gap.

9.2  ANTI-BAYESIAN CONFIDENCE ESCALATION
──────────────────────────────────────────

A June 2025 arXiv preprint (arXiv:2505.19184, under review) documented
a previously undescribed failure mode: in multi-agent debates, LLM agents'
confidence actively escalates from 72.9% at opening to 83.3% at closing
rounds — the opposite of rational Bayesian updating. In 61.7% of observed
debates, both sides simultaneously claimed ≥75% win probability, a
mathematical impossibility.

ARISTOTLE's Confidence Escalation Guard (in monitor.py) hides agent
self-confidence from peers, uses external C-DAG Bayesian aggregation
bypassing agent self-reports, and triggers calibration reset when
escalation is detected. This directly blocks the documented failure mode.

9.3  THE NL-TO-DEBATE-TOPOLOGY GAP
────────────────────────────────────

NL2Workflow (NAACL 2025) translates natural language into service
composition workflows over fixed operator libraries. AFlow (ICLR 2025)
automates workflow generation via Monte Carlo Tree Search over predefined
operators. EvoAgentX (EMNLP 2025) evolves agentic workflows. All operate
over fixed, pre-defined operator libraries.

None perform open-ended debate topology construction — dynamically
inventing an agent team from scratch with calibrated epistemic priors
and information-theoretic stopping criteria from a natural language claim.
This capability is absent from the literature.

9.4  THE ACCESSIBILITY CHASM
──────────────────────────────

Industry data from 2025 is unambiguous:
  - Deloitte: 80% of orgs claim mature basic automation; only 28% say
    the same about agentic AI — a 52-point gap
  - MindStudio 2026: only 2% of organizations deployed AI agents at scale
  - Gartner: over 40% of agentic AI projects projected to be cancelled
    by 2027 — not because models fail, but due to configuration complexity
  - Konsulteer 2025: half of organizations running fewer than five
    agentic AI projects total

ARISTOTLE is the first debate AI system designed from the ground up
for non-technical users.

9.5  ADAPTIVE HETEROGENEOUS TEAM CONSTRUCTION
───────────────────────────────────────────────

The A-HMAD paper (Springer 2025) confirmed that "existing debate methods
use homogeneous agents with simple majority voting, limiting their
effectiveness" and proposed diverse specialized agents. However, A-HMAD
still requires the researcher to manually define the agent heterogeneity
schema. topology.py answers the question A-HMAD leaves open: who decides
what heterogeneity is appropriate, and how is that decision automated?

9.6  LIVE ARGUMENT GRAPH VISUALIZATION
───────────────────────────────────────

No existing debate AI system provides live, event-by-event visual
rendering of debate progress as a structured argument DAG. ARGUS's
existing argus-viz provides excellent post-hoc analysis. AutoGen provides
text logs. METAL and MDebate output JSON. But the combination of
autonomous execution with concurrent, live-updating, layered DAG
visualization — where each argument node triggers immediate graph
rendering — does not exist in any current system.

9.7  CONVERSATIONAL ORCHESTRATION INTERFACE
────────────────────────────────────────────

Every current multi-agent framework uses either a code interface (Python
API) or a static form-based UI for orchestration. No system provides a
conversational chat interface where the user can interrogate the
orchestrator, ask it to explain decisions, or request topology adjustments
in natural language. ARISTOTLE's left chat pane is a first-of-its-kind
conversational orchestration layer.

9.8  GOVERNANCE OF THE ORCHESTRATOR ITSELF
──────────────────────────────────────────

The Springer/arXiv survey on agentic AI architecture (January 2026)
identified a meta-governance gap: "an open problem is how to combine
coordinated workflow automation with reliable and interpretable
decision-making, allowing broader autonomy without increasing the risk
of silent failure." No current orchestration system — not ServiceNow AI
Control Tower, not AWS Bedrock multi-agent collaboration, not IBM Watsonx
Orchestrate — provides a decision audit trail for the orchestrator layer
itself. ARISTOTLE's decision_log.py, which writes ORCHESTRATOR_DECISION
events to ARGUS's existing ProvenanceLedger, is a first-of-its-kind
response to this gap.

11.2  THE QUERY
────────────────

The user types in the left chat input:
  "Is social media actually causing the mental health crisis in teenagers,
   or is that just a moral panic?"

ARISTOTLE's typing indicator (animated "...") appears for 3-4 seconds.
Then ARISTOTLE responds in the left chat:

  "I have read your question. Here is how I understand it:

   You are asking whether there is a genuine causal relationship between
   social media use and adolescent mental health deterioration — or whether
   current public concern exceeds what the evidence actually supports.

   This is a causal claim with embedded methodological skepticism.

14.  CLOSING VISION
================================================================================

The landscape of agentic AI in 2025-2026 is characterized by extraordinary
capability and extraordinary inaccessibility existing simultaneously in the
same systems. Gartner predicts that over 40% of agentic AI projects will be
cancelled by 2027 — not because the models fail, but because organizations
cannot deploy and govern them at scale. Only 2% of organizations achieved
scale with AI agents in 2025.

ARGUS is already ahead of this curve in capability. ARISTOTLE — as a module
inside argus-debate-ai — is how it gets ahead of this curve in accessibility
and in doing so becomes a genuinely transformative tool rather than an
impressive but narrow research instrument.

The architectural decisions in this edition are not stylistic choices. Each
addresses a documented research gap or a confirmed failure mode:

  - The module placement inside argus-debate-ai means ARISTOTLE has zero
    installation friction — it comes with the package every ARGUS user
    already has

  - The single split-pane interface means the user never loses sight of
    the live debate while conversing with ARISTOTLE — the visualization
    and the conversation are always simultaneously present

  - The Full Lifecycle DAG makes the complete epistemic structure of the
    debate visible in one unified picture — Proposition at the top,
    Verdict at the bottom, every argument, challenge, and belief update
    in between, growing in real time

  - The anti-Bayesian confidence guard addresses a documented failure mode
    that most existing debate AI systems do not even know they have

  - The ORCHESTRATOR_DECISION audit trail solves the meta-governance
    problem: the orchestrator that makes consequential decisions is itself
    governed, logged, and auditable

When a user watches the cyan diamond Proposition node appear, watches
blue Specialist squares branch outward, watches green and red evidence
circles populate the middle tiers, watches amber rebuttals challenge the
strongest claims, watches lavender Bayesian Update circles accumulate the
weight of evidence round by round — and then watches the gold star Verdict
fill in at the bottom, pulsing once to signal that structured reasoning has
reached a conclusion — they are watching something that has never been
built before.

That is the version of ARGUS worth building. And it begins with:

  pip install argus-debate-ai
  argus aristotle run
